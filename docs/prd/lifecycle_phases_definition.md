# DevSquad V3.5 项目全生命周期定义

> **状态**: v3 已通过7角色评审（9条建议已采纳）
> **日期**: 2026-05-01
> **背景**: V2 时代定义了5角色8阶段，V3.5 升级为7角色体系（新增安全专家、运维专家），需要重新定义全生命周期阶段模型

---

## 一、设计原则

| # | 原则 | 说明 |
|---|------|------|
| 1 | 阶段细，不遗漏 | 细粒度阶段，每个阶段职责单一，必要时调用（如数据设计、交互设计可跳过） |
| 2 | 依赖驱动并行 | 无依赖关系的阶段可并行，有依赖的必须串行 |
| 3 | 依赖物/产出物明确 | 每阶段定义输入（依赖物）和输出（产出物），缺依赖物不可进入 |
| 4 | 门禁强制 | 门禁条件强制执行，不达标必须说明差距和原因，由用户决定是否推进 |
| 5 | 计划-执行闭环 | 测试计划定义"测什么"，测试执行按计划"全量执行" |
| 6 | 需求变更可控 | 任何阶段都可发起需求变更，但必须经过影响分析和评审 |
| 7 | 动态拼装+阶段模板 | 代码层面保留动态拼装能力，同时提供标准阶段模板供组合 |

---

|---|------|--------|---------|--------|--------------|--------------|---------|------|
| P1 | 需求分析 | Requirements Analysis | pm | arch+test+sec+ui | 用户原始需求 | 用户故事、验收标准、优先级矩阵、非功能性需求 | 验收标准可量化、无歧义 | ❌ 必选 |
| P2 | 架构设计 | Architecture Design | arch | pm+sec+infra | P1产出物 | 架构方案、技术选型、服务边界、质量属性决策 | 架构方案通过加权共识 | ❌ 必选 |
| P3 | 技术设计 | Technical Design | arch+coder | coder+test | P2产出物 | API规范、技术约束、接口定义、技术风险评估 | API规范无歧义 | ❌ 必选 |
| P4 | 数据设计 | Data Design | arch+coder | arch+sec | P2产出物+P3产出物(如存在) | 数据模型(ER图)、表结构、索引策略、迁移方案 | 数据模型3NF或反范式决策有据 | ✅ 可选 |
| P5 | 交互设计 | Interaction Design | ui | pm+test+sec | P1产出物+P3产出物 | 交互流程图、信息架构、原型、无障碍清单 | 核心流程可用性验证通过 | ✅ 可选 |

| P11 | 运维保障 | Operations & Assurance | infra+sec | arch+infra | P10产出物 | 监控大盘、告警规则、应急预案、性能基线 | P99<目标值、告警覆盖率100% | ✅ 可选 |

### 2.2 阶段详解

#### P1 需求分析 — 产品经理主导

**目标**：将模糊的业务诉求转化为可执行的需求规格

**依赖物**：用户/老板/客户的原始需求描述
**产出物**：
- 用户故事（As a... I want... So that...）
- 验收标准（Given/When/Then）
- 优先级矩阵（MoSCoW：Must/Should/Could/Won't）
- 非功能性需求（性能、安全、可用性指标）

**门禁**：所有验收标准可量化、无歧义；产品经理签字确认
**评审人**：arch（技术可行性）+ test（可测试性）+ sec（安全需求覆盖）+ ui（交互可行性）
**门禁不达标时**：列出哪些验收标准不可量化或存在歧义，由用户决定是否推进

**典型场景**：
- 创业公司拿到投资，需要将愿景转化为PRD
- 客户说"做一个管理系统"，PM拆解为具体模块+优先级
- 既有系统加新功能，PM评估影响范围和用户价值

---

#### P2 架构设计 — 架构师主导

**目标**：确定系统整体结构和技术方向

**依赖物**：P1产出物（需求规格）
**产出物**：
- 架构方案（单体/微服务/Serverless + 理由）
- 技术选型（语言/框架/中间件 + 选型依据）
- 服务边界（模块划分、服务间通信协议）
- 质量属性决策（性能目标、可用性目标、扩展性策略）

**门禁**：架构方案通过加权共识（≥70%），架构师和产品经理达成一致
**评审人**：pm（业务对齐）+ sec（安全架构）+ infra（部署可行性）
**门禁不达标时**：列出反对意见和分歧点，由用户裁决

**典型场景**：
- 从零搭建系统，评估单体vs微服务vs Serverless
- 系统需支持10万QPS，设计缓存层+消息队列+分库分表
- 多团队协作项目，定义服务边界和通信协议

---

#### P3 技术设计 — 架构师+开发者

**目标**：将架构方案细化为可开发的技术规格

**依赖物**：P2产出物（架构方案）
**产出物**：
- API规范（OpenAPI/Swagger、错误码、版本策略）
- 接口定义（服务间接口、第三方接口）
- 技术约束（编码规范、分支策略、依赖管理）
- 技术风险评估（可行性验证结果、备选方案）

**门禁**：API规范无歧义
**评审人**：coder（可实现性）+ test（可测试性）
**门禁不达标时**：列出哪些接口定义不明确，由用户决定是否推进

**典型场景**：
- RESTful API规范设计（URL/HTTP方法/状态码/分页）
- WebSocket实时推送接口定义
- API版本管理策略

---

#### P4 数据设计 — 架构师+开发者（可选）

**目标**：设计数据存储方案

**依赖物**：P2产出物（架构方案）+ P3产出物（如存在，可参考API规范设计字段结构）— 可与P3并行
**产出物**：
- 数据模型（ER图、表结构）
- 索引策略（查询场景→索引设计）
- 迁移方案（初始建表、数据迁移、版本管理）
- 数据安全策略（加密字段、脱敏规则、备份策略）

**门禁**：数据模型3NF或反范式决策有据
**评审人**：arch（架构一致性）+ sec（数据安全）
**门禁不达标时**：列出范式违规项及理由，由用户决定是否推进

**典型场景**：
- 电商系统设计订单/商品/用户核心数据模型
- 多租户SaaS设计数据隔离方案（共享DB/独立Schema/独立DB）
- 历史数据归档策略，冷热数据分离方案

> **调用条件**：涉及数据库/持久化存储的项目必选，纯前端/工具类项目可跳过

---

#### P5 交互设计 — UI设计师主导（可选）

**目标**：确保技术方案对用户友好

**依赖物**：P1产出物（需求规格）+ P3产出物（技术约束）
**产出物**：
- 交互流程图
- 信息架构
- 响应式布局方案
- 无障碍合规检查清单

**门禁**：核心流程可用性验证通过
**评审人**：pm（业务对齐）+ test（可测试性）+ sec（隐私合规）
**门禁不达标时**：列出可用性问题，由用户决定是否推进

**典型场景**：
- 新功能开发前，设计交互流程和信息架构
- 用户反馈操作复杂，简化流程减少步骤
- 无障碍合规审查（色彩对比度、键盘导航、屏幕阅读器）

> **调用条件**：面向终端用户的项目必选，纯后端/内部工具可跳过

---

#### P6 安全评审 — 安全专家主导（可选）

**目标**：确保方案不引入安全风险

**依赖物**：P2+P3产出物（P4、P5产出物如存在也纳入评审）
**产出物**：
- 威胁模型（STRIDE分析）
- 漏洞清单（OWASP Top 10检查）
- 合规报告（GDPR/CCPA/HIPAA/PCI-DSS）
- 安全修复建议（按P0/P1/P2分级）

**门禁**：无P0/P1漏洞；合规项全绿；安全专家签字
**评审人**：arch（安全架构合理性）+ infra（安全运维可行性）
**门禁不达标时**：列出P0/P1漏洞和合规风险，由用户决定是否承担风险推进

**典型场景**：
- 上线前安全扫描，检查OWASP Top 10
- 数据合规审查（GDPR/CCPA/HIPAA）
- 第三方依赖安全评估，检查已知CVE

> **调用条件**：涉及任何敏感数据、对外暴露、或合规要求的项目必选，仅当"无任何敏感数据、无对外暴露、无合规要求"时可跳过
> **否决权**：安全专家可否决带P0漏洞的版本上线

---

#### P7 测试计划 — 测试专家主导

**目标**：在开发前系统规划所有测试维度，确保"先规划后执行"

**依赖物**：P1产出物（验收标准）+ P3产出物（技术设计）+ P6产出物（安全评审，如存在）
**产出物**：

| 测试维度 | 计划内容 | 协作角色 | 是否必选 |
|----------|---------|---------|---------|
| **功能测试** | 测试用例设计、边界值分析、等价类划分 | coder | ✅ 必选 |
| **集成测试** | 模块间接口测试、第三方服务Mock、数据流验证 | arch, coder | ✅ 必选 |
| **性能测试** | 基准测试、负载测试、压力测试、稳定性测试的指标和方案 | arch, infra | 🔍 P7评审决定 |
| **安全测试** | 渗透测试范围、漏洞扫描策略、合规验证方法 | sec | 🔍 P7评审决定 |
| **环境依赖** | 测试环境规格、数据准备方案、环境隔离策略 | infra | ✅ 必选 |
| **安装手顺** | 安装/升级/回滚的验证步骤、兼容性矩阵 | infra | 🔍 P7评审决定 |
| **回归策略** | 回归测试范围、自动化率目标、CI门禁规则 | coder | ✅ 必选 |
| **验收标准** | 对应P1验收标准的逐条验证方法 | pm | ✅ 必选 |

**门禁**：测试计划评审通过
**评审人**：arch（技术可行性）+ sec（安全覆盖）+ infra（环境就绪）+ pm（验收覆盖）
**门禁不达标时**：列出评审不通过项和修改建议，由用户决定是否推进

**典型场景**：
- 电商系统：规划支付模块的集成测试（对接支付网关）、性能测试（秒杀场景10万QPS）、安全测试（支付数据加密验证）
- SaaS平台：规划多租户隔离测试、数据迁移兼容性测试、安装升级手顺验证
- 内部工具：P7评审决定跳过性能测试和安全测试，仅保留功能/集成/回归

> **关键设计**：P7评审决定哪些测试维度需要执行。小项目可在评审中跳过性能测试、安全测试等维度，但必须明确记录"跳过原因"。

---

#### P8 开发实现 — 开发者主导

**目标**：将设计转化为可运行的代码

**依赖物**：P3产出物（技术设计）+ P6产出物（安全修复建议，如存在）+ P7产出物（测试计划）
**产出物**：
- 可运行代码
- 代码审查报告
- 单元测试（覆盖率≥80%）
- 可测试性设计说明（为P9提供测试入口）

**门禁**：代码审查通过、无P0缺陷、单元测试全绿
**评审人**：arch（架构遵循）+ sec（安全编码）+ test（可测试性）+ coder（修复可行性）
**门禁不达标时**：列出P0缺陷清单和代码审查问题，由用户决定是否推进

**典型场景**：
- 架构师确定微服务后，开发者选择框架实现业务逻辑
- 代码审查检查规范/模式/错误处理/性能热点
- 性能优化：分析慢查询、修复N+1、引入缓存

> **注**：开发过程中需关注P7测试计划中的可测试性要求，如预留测试接口、Mock点、健康检查端点等

---

#### P9 测试执行 — 测试专家主导

**目标**：按P7测试计划系统执行全部测试维度，确保质量达标

**依赖物**：P7产出物（测试计划书）+ P8产出物（可运行代码）
**产出物**：

| 测试维度 | 执行内容 | 通过标准 |
|----------|---------|---------|
| **功能测试** | 执行全部测试用例、边界值验证 | 用例通过率100% |
| **集成测试** | 模块间接口调用、第三方服务联调 | 接口测试全通过 |
| **性能测试** | 基准/负载/压力/稳定性测试 | P99<目标值、无内存泄漏 |
| **安全测试** | 渗透测试、漏洞扫描、合规验证 | 无P0/P1漏洞 |
| **环境验证** | 测试环境与生产环境一致性验证 | 环境差异已记录且可控 |
| **安装验证** | 安装/升级/回滚手顺执行验证 | 手顺可执行、无遗漏 |
| **回归测试** | 全量回归或受影响范围回归 | 回归通过率100% |
| **验收验证** | 逐条验证P1的验收标准 | 验收标准100%满足 |

**门禁**：
- 单元测试覆盖率≥80%
- P7测试计划100%执行（P7评审决定执行的维度，每个维度都有执行记录和结果）
- 无P0缺陷
- P1验收标准逐条通过

**门禁不达标时**：列出未达标的维度、差距值、原因分析，由用户决定是否推进
**评审人**：arch（技术验证）+ pm（验收确认）+ sec（安全验证）+ infra（环境/安装验证确认）

**典型场景**：
- 支付模块：执行集成测试（对接支付网关）、性能测试（秒杀场景）、安全测试（加密验证）、安装手顺验证
- 内部工具：仅执行功能/集成/回归（P7评审已跳过性能/安全维度）

---

#### P10 部署发布 — 运维专家主导

**目标**：安全可靠地将系统推上生产环境

**依赖物**：P9产出物（测试通过报告）
**产出物**：
- 部署方案（容器化+编排）
- 发布策略（蓝绿/金丝雀/滚动）
- 回滚预案
- 发布清单（检查项）
- 基础设施定义（IaC模板、资源配置）

**门禁**：部署演练通过、回滚预案验证、运维专家签字
**评审人**：arch（部署架构）+ sec（上线安全）+ test（发布验证）
**门禁不达标时**：列出演练失败项和回滚风险，由用户决定是否推进

**典型场景**：
- Docker容器化 + Kubernetes编排
- 蓝绿部署/金丝雀发布策略设计
- 数据库迁移：评估停机时间、设计灰度切换

---

#### P11 运维保障 — 运维+安全（可选）

**目标**：确保系统在生产环境稳定运行

**依赖物**：P10产出物（部署完成）
**产出物**：
- 监控大盘（Prometheus + Grafana）
- 告警规则和升级策略
- 应急预案和演练计划
- 性能基线报告

**门禁**：P99<目标值、告警覆盖率100%、应急预案演练通过
**评审人**：arch（性能基线）+ infra（运维成熟度）
**门禁不达标时**：列出监控盲区和性能差距，由用户决定是否推进

**典型场景**：
- 建立监控大盘和告警规则
- 故障应急预案和定期演练
- 成本优化：分析资源使用率、提出缩容方案

> **调用条件**：生产环境长期运行的项目必选，一次性脚本/实验性项目可跳过

### 2.3 阶段依赖关系与并行

```
P1 需求分析 ──→ P2 架构设计 ──┬──→ P3 技术设计 ──→ P6 安全评审 ──→ P7 测试计划 ──→ P8 开发实现 ──→ P9 测试执行 ──→ P10 部署发布 ──→ P11 运维保障
                               │                        ↑              ↑                                    ↑
                               ├──→ P4 数据设计 ────────┤              │                                    │
                               │    (与P3并行)           │              │                                    │
                               └──→ P5 交互设计 ────────┘              │                                    │
                                    (依赖P1+P3)                         │                                    │
                                                                        └── P7产出物 ────────────────────────┘
                                                                            (测试计划指导P9执行)
```

**并行规则**：
- P4 与 P3 可并行（都只依赖P2）
- P5 在P3完成后可启动（依赖P1+P3）
- P6 需等P3完成（+P4/P5如存在）
- P7 需等P6完成（如P6存在）或P3完成（如P6跳过）

**可选阶段跳过规则**：

| 阶段 | 跳过条件 | 跳过时的替代 |
|------|---------|------------|
| P4 数据设计 | 纯前端/工具类项目，无持久化存储 | P3中简单定义数据格式 |
| P5 交互设计 | 纯后端/内部工具，无终端用户 | 无 |
| P6 安全评审 | 无任何敏感数据、无对外暴露、无合规要求 | P7中保留基本安全测试维度 |
| P11 运维保障 | 一次性脚本/实验性项目 | P10中包含基本健康检查 |

## 三、需求变更流程

任何阶段均可触发需求变更，但必须经过影响分析和评审：

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ 变更发起  │───→│ 影响分析  │───→│ 变更评审  │───→│ 批准/驳回 │───→│ 回退执行  │
│ pm/user  │    │arch+sec  │    │ 全角色    │    │  pm+arch │    │ 受影响阶段│
│          │    │ +test    │    │          │    │          │    │          │
└──────────┘    └──────────┘    └──────────┘    └──────────┘    └──────────┘
     │               │               │               │               │
     │ 发起人填写     │ 评估影响范围   │ 全角色评审     │ pm+arch决策    │ 回退到最早
     │ 变更内容+理由  │ 和受影响阶段   │ 权衡利弊       │               │ 受影响阶段
     │               │ 评估工作量     │               │               │ 重新执行
```

### 3.1 变更发起

**谁可以发起**：用户、产品经理
**何时可以发起**：任何阶段
**发起内容**：
- 变更描述（改什么）
- 变更理由（为什么改）
- 期望时间（什么时候要）

### 3.2 影响分析

**分析人**：arch（技术影响）+ sec（安全影响）+ test（测试影响）
**分析内容**：

| 维度 | 分析项 |
|------|--------|
| 技术影响 | 哪些模块受影响、是否需要架构调整 |
| 安全影响 | 是否引入新的安全风险 |
| 测试影响 | 哪些测试需要重跑、回归范围 |
| 进度影响 | 预估额外工作量、对里程碑的影响 |
| 受影响阶段 | 需要回退到哪个阶段重新执行 |

### 3.3 变更评审

**评审人**：全角色（7角色）
**评审方式**：加权共识
**评审结论**：
- ✅ 批准：回退到受影响阶段，重新执行
- ❌ 驳回：维持原方案，记录驳回理由
- 🔄 修改：调整变更方案后重新评审

### 3.4 回退规则

| 当前阶段 | 需求变更影响 | 回退到 |
|----------|------------|--------|
| P2 架构设计 | 需求范围变化 | P1 需求分析 |
| P3 技术设计 | 架构方案变化 | P2 架构设计 |
| P4 数据设计 | 数据模型变化 | P4 重新设计 |
| P5 交互设计 | 交互流程变化 | P5 重新设计 |
| P6 安全评审 | 安全风险变化 | P6 重新评审 |
| P7 测试计划 | 测试范围变化 | P7 重新计划 |
| P8 开发实现 | 功能/接口变化 | P3 或 P8 |
| P9 测试执行 | 测试范围变化 | P7 或 P9 |
| P10 部署发布 | 部署方案变化 | P10 重新部署 |
| P11 运维保障 | 监控/告警变化 | P11 重新配置 |

## 四、门禁机制

### 4.1 门禁原则

- **强制执行**：每个阶段的门禁条件必须检查
- **不达标不阻塞**：门禁不达标时，不自动阻塞流程，而是**生成差距报告**
- **用户决策**：差距报告包含"差距项+原因分析"，由用户决定是否推进
- **留痕**：所有门禁检查结果（通过/不达标+用户决策）记录到检查点

### 4.2 差距报告格式

```json
{
  "phase": "P9",
  "gate_check": {
    "unit_coverage": {"target": "80%", "actual": "75%", "status": "FAIL"},
    "plan_execution": {"target": "100%", "actual": "100%", "status": "PASS"},
    "p0_defects": {"target": 0, "actual": 0, "status": "PASS"}
  },
  "gaps": [
    {
      "item": "unit_coverage",
      "gap": "5% below target",
      "reason": "3 utility modules lack tests",
      "recommendation": "Add tests before P10, or accept risk"
    }
  ],
  "user_decision": "PENDING"
}
```
