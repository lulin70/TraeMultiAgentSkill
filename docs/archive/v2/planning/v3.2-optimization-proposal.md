# DevSquad v3.2 优化提案 — 多角色协作审核

> **日期**: 2026-04-16
> **状态**: 🔄 待多角色共识评审
> **触发**: v3.1 交付后战略审视，识别出 4 个关键差距 + 6 个候选优化方向

---

## 1. 现状回顾

### 1.1 v3.1 已达成能力

| 维度 | 数据 |
|------|------|
| 核心模块 | **15 个** (Coordinator/Worker/Scratchpad/Consensus/BatchScheduler/Compressor/PermissionGuard/Skillifier/Warmup/MemoryBridge/Dispatcher/TestQualityGuard/PromptAssembler/VariantGenerator) |
| 测试覆盖 | **~782 / ~782 PASS (100%)** |
| 架构目标达成 | **6/6 核心指标全部达标 + 2项超额** |

### 1.2 识别出的 4 个关键差距

#### 差距 A: Worker 是"空壳" —— 缺少真正的执行引擎
- Worker.execute() 只产出 prompt 文本，不实际调用 LLM/API
- 当前依赖 Trae 作为外部执行层
- 集成 OpenClaw/Claude Code 时需补上此层

#### 差距 B: 缺少端到端 Demo / 真实场景验证
- ~782 个单元测试，但从未串联跑过完整流程
- 无法向用户展示系统真正价值
- 无法发现跨组件集成的隐性 bug

#### 差距 C: Dispatcher "quick_dispatch" 体验不够人性化
- 只是 dispatch() 的薄包装
- 输出报告格式不够结构化

#### 差距 D: 缺少学习效果的可观测性
- MemoryBridge/Skillifier/VariantGenerator 各有统计
- 无统一聚合视图

## 2. 候选优化方向（6 项）

### 方向 ①: E2E Demo 脚本（端到端演示）

**描述**: 编写一个完整的端到端 Demo，用真实任务跑通全链路

**范围**:
```
用户输入自然语言任务
  → Dispatcher.analyze_task() 自动识别角色
  → Coordinator.plan_task() 分解子任务
  → BatchScheduler.schedule() 并行调度
  → Worker._do_work() × N 各自产出
  → Scratchpad 共享发现/检测冲突
  → ConsensusEngine 投票解决冲突
  → Coordinator.generate_report() 输出报告
  → MemoryBridge 记录经验
  → Skillifier 提取模式
```

**投入**: ~250 行代码 + 1 个测试脚本
**ROI**: ⭐⭐⭐⭐⭐ (验证全系统 + 最佳演示)
**风险**: 低（纯集成层）
**依赖**: 无

---

### 方向 ②: ExecutionEngine 执行引擎

**描述**: 让 Worker 真正能独立执行任务，不再依赖外部调用方

**核心接口**:
```python
class ExecutionEngine:
    def execute_instruction(self, instruction: str,
                            context: Dict) -> ExecutionResult:
        """执行 Worker 产出的指令，返回结构化结果"""
        # 支持: LLM调用 / 工具调用 / 文件操作 / API请求

    def register_tool(self, tool_name: str, handler: Callable):
        """注册可执行工具"""
```

**投入**: ~350 行代码 + 接口抽象层
**ROI**: ⭐⭐⭐⭐ (解耦对Trae的依赖，为OpenClaw/Claude Code铺路)
**风险**: 中（需定义工具协议）
**依赖**: 方向①（Demo 可复用 Engine）

---

### 方向 ③: LearningDashboard 学习仪表盘

**描述**: 统一聚合 MemoryBridge + Skillifier + VariantGenerator 的学习成果

**展示内容**:
- 总记忆条目数 / 按类型分布 / 召回命中率
- 提取的成功模式数 / 晋升的提示词变体数
- 学习效果趋势图（随时间变化）
- Top-5 最有价值的学习发现

**投入**: ~180 行代码 + Markdown 模板
**ROI**: ⭐⭐⭐ (可观测性提升，方便调优)
**风险**: 低
**依赖**: 无

---

### 方向 ④: Dispatcher UX 增强

**描述**: 提升 quick_dispatch 的输出质量

**改进点**:
- 结构化 Markdown 报告（含：任务摘要 / 角色分工 / 关键发现 / 冲突解决 / 下一步建议）
- 执行时间线可视化
- Token 使用统计
- 一键导出为 SKILL.md 格式

**投入**: ~100 行代码
**ROI**: ⭐⭐⭐ (用户体验直接提升)
**风险**: 低
**依赖**: 无

---

### 方向 ⑤: MCE 集成（memory-classification-engine）

**描述**: 接入外部记忆分类引擎，提升 MemoryBridge 的精度

**已规划状态**: Phase 5（v3-upgrade-proposal.md 中已记录4阶段路线图）

**条件前置**: MCE 项目达到稳定版（可 pip install）

**投入**: ~300 行代码（4个集成点）
**ROI**: ⭐⭐⭐⭐ (记忆召回噪声 ↓60%)
**风险**: 中高（依赖外部项目稳定性）
**依赖**: MCE 项目发布稳定版

---

### 方向 ⑥: 跨平台适配层

**描述**: 统一 OpenClaw / Claude Code / Trae 的接口差异

**核心思路**:
```python
class PlatformAdapter(ABC):
    @abstractmethod
    def call_llm(self, prompt: str) -> str: ...
    @abstractmethod
    def read_file(self, path: str) -> str: ...
    @abstractmethod
    def write_file(self, path: str, content: str): ...
    @abstractmethod
    def search_code(self, query: str) -> List[SearchResult]: ...

class TraeAdapter(PlatformAdapter): ...
class ClaudeCodeAdapter(PlatformAdapter): ...
class OpenClawAdapter(PlatformAdapter): ...
```

**投入**: ~250 行代码 + 适配器实现
**ROI**: ⭐⭐⭐⭐ (一套代码多平台运行)
**风险**: 中（各平台API差异可能超预期）
**依赖**: 方向②（Engine 需要 Adapter）

## 4. 审核要求

请各角色从以下维度评审以上 6 个方向：

### 产品经理 (PM)
- 用户价值判断：哪个方向对用户最有用？
- 优先级排序：如果资源有限，先做哪些？
- MVP 定义：最小可用版本应包含哪些？

### 架构师 (Architect)
- 架构合理性：方向②和⑥的设计是否合理？
- 技术债务：是否会引入新的耦合？
- 扩展性：为未来留了足够的扩展空间吗？

### UI设计师 (UI Designer)
- ③ LearningDashboard 的信息架构建议
- ④ Dispatcher 输出报告的视觉层次建议
- 整体用户体验流畅性评估

### 独立开发者 (Solo Coder)
- 实现可行性：各方向的工程量评估是否准确？
- 测试策略：如何保证新功能的质量？
- 代码复用：能否最大化利用现有模块？

### 测试专家 (Tester)
- ① E2E Demo 的测试覆盖策略
- 各方向的风险点和边界条件
- 回归测试影响评估

---

## 5. 决策机制

1. 各角色独立提交评审意见（写入 Scratchpad）
2. 如有分歧 → ConsensusEngine 投票
3. 最终产出 **v3.2 Roadmap 共识决议**
